* Initialisation of breeze
:PROPERTIES:
:ID:       156b2ecb-b87b-4ef8-b60b-e688d5da4bc1
:END:

There are multiple things that can trigger breeze's initialization:
- slime or sly is connected (via ~slime-connected-hook~ and ~sly-connected-hook~)
- one of breeze's command is called (e.g. ~breeze-capture~)
- ~breeze-init~ is called

The initilisation is a bit complex, here's the high-level view:

1. wait for the listener to be connected
2. evaluate form(s) to check if breeze is already loaded
3. evaluate form(s) to load breeze's code
4. wait for breeze's code to be loaded

I have 2 problems:
1) the "wait for listener to be connected" is sligtly flaky, the
   predicate I use _can_ return true before the listener is actually
   ready to receive things to evaluate (I think).
2) ~breeze-init~ and ~breeze-ensure~ can be called multiple time
   concurrently

N.B. emacs does provide mutexes and condition variables.

** Problem 1)

Problem 1) looks like this:

#+begin_src text
Polling "/tmp/slime.1" .. 6 (Abort with ‘M-x slime-abort-connection’.)
Breeze: waiting for listener to be connected...
Polling "/tmp/slime.1" .. 7 (Abort with ‘M-x slime-abort-connection’.)
Connecting to Swank on port 43535..
Breeze: waiting for listener to be connected...
; pipelined request... (cl:eval (cl:and (cl:or (cl:find-package "breeze.utils")
Loading breeze’s system asynchronously...
error in process filter: Reply to canceled synchronous eval request tag=slime-result-3-1 sexp=(swank:swank-require '(swank-indentation swank-trace-dialog swank-package-fu swank-presentations swank-macrostep swank-
fuzzy swank-fancy-inspector swank-c-p-c swank-arglists swank-repl))
#+end_src

I'm not 100% certain this happens because the listener is not actually
ready...

*** Possible solution: wait for *-connected-hook

I _think_ sly and slime's *-connected-hook is only invoked when the
repl is actually ready...

Might require a per-connection mutex + condition variable...
1. wait for any connection
2. for this connection, wait for the "connected-hook"??

*** Possible solution: load breeze as a sly/slime contrib

*** Other notes

I checked and (in slime at least), the "connected-hook" is called
after the contribs are loaded

oh! I might be able to wait for the variable
~slime-inferior-lisp-connected~ to be set.

*** Resolution

ok... I think I fixed the issue by adding a call to ~slime-sync~ or
~sly-sync~ just after ~breeze-wait-for-listener~.

BUT, this was a flaky bug, so maybe I was just lucky and didn't
reproduce it yet...

** Problem 2)

One of problem 2) 's symptoms is that user sees a lot of "Waiting for
breeze to be initialized...". Normally, they would see 2 per seconds,
but they would see 2 per seconds, per invocation of ~breeze-init~.

I'm not sure if there are any adverse effect on the loading of breeze
itself...

It would definitely invoke ~breeze-refresh-commands~ multiple time,
which is very probably a huge waste of time... but might not cause any
other issues...

~breeze-init~ is called by the "connected hook", via
~breeze-connected-hook-function~.

*** Possible solution: per-connection "breeze-initialized-p"

This has one very annoying problem: it would need another mutex to
make sure that the per-connection mutex is initialized only once...

*** Resolution

For now... I'll just remove the message "Waiting for breeze to be
initialized..." (that I _just_ added), I'll come back to this issue if
I see some other bug that could be caused by this race condition.
