#+title: Testing Breeze

#+begin_quote
This page is about breeze's development. It is not about using breeze
for testing.
#+end_quote

* Summary of all the tests

- common lisp unit tests
  - ~make test~
  - for each common lisp implementation:
    - ~guix-shell-test-*~
    - ~guix-container-test-*~
- emacs lisp unit tests
- emacs integration tests
- publish the org-mode documents
  - ~make public~
- build the executable

* How to run the tests

#+begin_src lisp
(ql:quickload "breeze/test")
(asdf:test-system "breeze")
#+end_src

Or from the command line:

#+begin_src shell
./scripts/test-sbcl.sh
#+end_src

OR

#+begin_src shell
make test
#+end_src

* Manual tests

There are tons of automated tests, but unfortunately, not everything
can be automated easily or not without a lot of time. So here are
some manual checks that can complement the automated tests.

** Testing the "initialisation" of emacs in emacs

- in emacs
  - don't start slime or sly
  - open a lisp file
  - see if breeze-minor-mode is enabled (~M-; breeze-minor-mode~)
  - if it is, there should be a header-line (line a mode-line, but at
    the top, and per-buffer)
  - if you run a "breeze command", like "quickfix", it should ask you
    to start a listener
  - start slime
  - check that the header-line is updated (it should show the current connection)
  - try a "breeze command" again
  - open another lisp file and check the header-line and run a command
    - I've had issues where the _first_ lisp file opened was not setup
      correctly, so opening 2 files helps finding these kind of
      issues.

** Review workbenches

There are two files =workbench.lisp= and =workbench.el= that contains
code that is only meaningful when run manually and piecewise.

Once in a while, review those files to make sure
1. they are not too messy
2. the content is not obsolete
3. the content is not duplicated (e.g. ~(trace x)~ in multiple places)
4. they are somewhat readable
5. they might contains TODOs, it might make sense to move those
   somewhere more visible
